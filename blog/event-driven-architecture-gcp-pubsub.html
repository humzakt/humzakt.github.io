Event-Driven Architecture on GCP: Pub/Sub, Cloud Tasks, and Cloud Scheduler in Practice

Feb 8, 2026 · 9 min read · Humza Tareen
GCP Pub/Sub Event-Driven Cloud Tasks Microservices

Why Event-Driven?

Our AI evaluation platform has a problem that REST APIs can't solve cleanly: a single evaluation task takes 5-30 minutes, touches 6 different services, and can fail at any point.

With synchronous HTTP: timeouts, retries hitting already-completed work, and debugging nightmares. With events: each service subscribes independently. If one fails, it retries from its own checkpoint. Others don't even know.

The Pipeline Phases

Phase A: Task Creation → (Pub/Sub: task-events)
Phase B: Preprocessing → (Pub/Sub: phase-b-batch-notifications)
Phase C: Agent Execution → (Pub/Sub: phase-c-asset-feed)
Phase D: Output Collection → (Pub/Sub: phase-e-jobs)
Phase E: Judging & Scoring → (Pub/Sub: phase-f-jobs)
Phase F: Aggregation & Learning → (Pub/Sub: evaluation-events)
Consumers: Notification, RAG Sync, Analytics

Fan-Out Pattern

Our most powerful pattern: one topic, multiple subscriptions. evaluation-events is published to once but consumed by notification service, RAG sync, analytics pipeline, and auto-eval scheduler. Adding a new consumer? Create a subscription. Zero changes to the publisher.

Dead-Letter Queues

Every production topic has a DLQ with a 5-retry policy. We have a weekly DLQ review. In 6 months, we caught 3 schema migration issues, 1 Firestore quota limit, and 2 transient Cloud SQL connection failures. Without DLQs, these would have been silent data loss.

Cloud Scheduler for Automation

Each scheduled job publishes a message to a Pub/Sub topic. The consuming service doesn't know if the message came from a schedule, a user action, or an API call. We could pause auto-eval-daily in production without touching any service code.

Monitoring

Every Cloud Run service has an alert at >5% error rate. We also monitor CPU utilization < 10% to auto-detect idle VMs — this saves ~40% on compute costs.

Key Lessons

  • Message ordering doesn't matter if you design correctly. Every message contains enough context to be processed independently.
  • Acknowledge after commit, not before. Write first, ack second, always.
  • Schema evolution is inevitable. Version your message schemas.
  • Pub/Sub for events, Cloud Tasks for commands. Keep the distinction clear.
