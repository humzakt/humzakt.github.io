Building a RAG Retrieval Service: pgvector, Embedding Migrations, and Provenance Tracking

Feb 8, 2026·6 min read·Humza Tareen
RAGpgvectorEmbeddingsPostgreSQLFastAPI

What This Service Does

The RAG Retrieval Service enables continuous learning via pgvector similarity search. I contributed 4 focused PRs, each solving a critical problem in production.

Emergency Embedding Migration (PR #7)

text-embedding-004 was deprecated overnight. All queries started returning 404 errors. This PR made the embedding model configurable via environment variables (EMBED_MODEL, EMBED_DIMENSIONS).

Migration strategy:

  • Side-by-side columns for rollback capability
  • Batch re-embedding pipeline
  • Feature flag switch for zero-downtime deployment

Result: 470 lines across 6 files, zero downtime migration. The service now supports any embedding model via configuration.

Provenance and Metadata (PR #4)

CLI eval agents needed outcome data for calibration. I added provenance and metadata options to the retrieve endpoint:

  • Updated SQL to include d.provenance and d.source_type
  • Optional return fields for bandwidth savings when not needed
  • Enables downstream services to track data lineage

Response Serialization Fix (PR #5)

The response included "provenance": null which was breaking downstream consumers. Fixed by setting response_model_exclude_none=True in the FastAPI endpoint.

Infrastructure

  • Database: Cloud SQL PostgreSQL 17 + pgvector extension
  • Index: IVFFlat index with cosine similarity (100 lists)
  • Embedding Model: text-embedding-3-large (768 dimensions)
  • Re-indexing: Daily Cloud Scheduler job for optimal performance
  • Deployment: Cloud Run
