I Audited Our AI Platform and Found 25 Critical Security Issues

Feb 8, 2026 · 7 min read · Humza Tareen
Security GCP Cloud Security Python DevOps

The Motivation

When you're shipping fast, security debt accumulates silently. I decided to audit our entire AI evaluation platform systematically across 10+ services.

Critical Findings

  1. SQL injection — BigQuery queries using f-strings instead of parameterized queries
  2. Hardcoded credentials in alembic.ini (database migration tool)
  3. Secrets in CI/CD logs — API keys printed during build steps
  4. Missing authentication on 3 internal endpoints (assumed "internal = safe")
  5. PII in application logs — user emails and evaluation data in plaintext logs

Architecture Issues

  • A 3,677-line ParallelOrchestrator class (textbook God Class)
  • V1 dependencies leaking into V2 code (architecture boundary violation)
  • In-memory rate limiting (doesn't work across Cloud Run instances)
  • CPU-bound algorithms blocking the async event loop

The Fixes

Security

  • Moved all secrets to GCP Secret Manager
  • Parameterized every database query
  • Added structured logging with PII stripping
  • Implemented authentication middleware on every endpoint
  • Added pre-commit hooks: ruff, type checking, security scanning

Architecture

  • Broke the God Class into 5 focused services
  • Built an LLM Gateway abstraction (swap providers without code changes)
  • Moved rate limiting to Redis (shared across instances)
  • Wrapped CPU-bound work in thread pools

The Filing System

156 GitHub issues with severity labels: 25 Critical, 50 High, 60 Medium, 21 Low. Systematic issue tracking meant nothing fell through the cracks.

The takeaway: schedule security audits. Don't wait for an incident. The 2 days I spent auditing saved us from at least 3 potential production incidents.
