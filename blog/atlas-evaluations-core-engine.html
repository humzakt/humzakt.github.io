Building the Core AI Evaluation Engine: 72 PRs, AlloyDB Fixes, and a Headless API

Feb 8, 2026·12 min read·Humza Tareen

What This Service Does

The Evaluation Engine is the brain of the entire AI assessment platform. It manages the full lifecycle of evaluating AI coding agents: creating tasks, orchestrating agent execution across GKE clusters, collecting outputs, running LLM-based judging, and aggregating results. I contributed 72 pull requests to this service over 4 months.

The AlloyDB SSL Crisis (PRs #264–#395)

This was a multi-week battle. Our evaluation tasks — which run 5-30 minutes — were randomly failing at a 5% rate with SSL connection has been closed unexpectedly.

The timeline of fixes:

  1. PR #264 — First attempt: retry agent-runs fetch on DB SSL disconnect (+608/-55). Added retry decorators. Helped, but didn't fix root cause.
  2. PR #267 — Suppress DB SSL errors on session close. Prevent noisy traces from masking the real problem.
  3. PR #268 — Firestore score verification + DB pool alignment (+378/-22). Discovered scores writing to the wrong Firestore database.
  4. PR #272 — Retry agent_runs on transient SSL disconnects. More targeted retry logic.
  5. PR #282 — Harden sessions + transient DB reads (+257/-40). Root cause analysis session.
  6. PR #287 — Reduce stale SSL pooled connections. Found actual root cause: AlloyDB kills idle connections at ~10 min, pool recycled at 15 min.
  7. PR #395 — Final fix: pool_recycle=180, pool_pre_ping=True (+164/-74).

Result: 5% failure rate → 0.01%. Seven PRs over 2 weeks to fully understand and fix one infrastructure bug.

The Firestore Default Database Trap (PRs #268, #271, #294, #300)

Firestore has a (default) database. If you don't specify which database to use, everything routes there. We had 12 Firestore databases, but several code paths were accidentally hitting the default.

PR #294 was the definitive fix: prevent Firestore default-db misrouting + restore auto-export. Every Firestore reference now explicitly specifies its database ID.

PR #300 extended this to the React frontend — real-time listeners were using the wrong database, causing stale UI data.

Continuous Learning Pipeline (PR #411)

PR #411 resolved gaps preventing agent training on historical performance. Fixed data flow between the evaluation engine, RAG service (pgvector), and auto-rater (+211/-9).

Headless API Mode (PR #415)

The final major feature: a headless API mode (+2,541 lines) allowing external services to trigger evaluations programmatically without the web UI. Teams can now run agent evaluations as part of their CI/CD pipelines.

KEDA Autoscaling (PR #306)

We use KEDA on GKE to scale agent execution pods based on Pub/Sub message backlog. PR #306 fixed an invalid idleReplicaCount setting preventing scale-to-zero, and warm replicas not being maintained. After the fix, the cluster properly scales 0→39 nodes based on evaluation queue depth.

Security Audit (37 Issues Filed)

  • Critical: Firestore rules allowing unauthenticated read/write, production K8s using :latest tags, zero frontend test coverage, no React Error Boundaries
  • High: Excessive any type usage (60+ instances), no code splitting, conflicting HPA and KEDA autoscaling
